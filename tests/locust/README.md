# Нагрузочное тестирование сервиса генерации упражнений

Инструмент эмпирически измеряет пропускную способность одного инстанса
`exgen` и распределение времени отклика `POST /api/v1/generate` при разных
уровнях конкурентности. Результаты предназначены для раздела ВКР про
тестирование и оценку ресурсоёмкости.

## Что измеряется

Для каждого уровня конкурентности (1, 2, 4, 8, 16 пользователей) считаются:

- **total_requests** — общее число запросов за прогон.
- **success_rate** — доля запросов, для которых сервис вернул HTTP 200 с
  непустым полем `exercise`. Все остальные ответы (HTTP ≠ 200, пустое тело,
  невалидный JSON, таймаут 120 с) считаются неудачными.
- **RPS** — измеренная пропускная способность (запросов в секунду).
- **latency_p50 / p95 / p99** — медиана и хвостовые перцентили времени
  отклика, в миллисекундах.
- **fallback_rate** — доля *успешных* ответов, в которых сработал шаблонный
  резервный путь (LLM вернула неразборный JSON, и сервис собрал упражнение
  правилами spaCy). Этот ответ всё ещё имеет HTTP 200 и непустой `exercise`,
  но `status == "fallback"` и заполнен `fallback_reason`.

Разделение «ok / fallback» важно: оба ответа полезны для клиента, но только
`ok` подтверждает, что LLM действительно отработала на этой нагрузке. При
росте конкурентности fallback_rate может расти, если декодирование начинает
сбоить под давлением — это самостоятельный сигнал, отдельный от RPS.

## Установка

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r tests/locust/requirements.txt
```

`locust` намеренно вынесен в отдельный `requirements.txt`, чтобы не утяжелять
основной образ сервиса.

## Запуск

Сервис должен быть уже поднят (локально через `docker compose` или удалённо).
Адрес задаётся переменной окружения `TARGET_HOST` (по умолчанию
`http://localhost:8000`):

```bash
# Локально
./tests/locust/run_benchmark.sh

# Против удалённого хоста
TARGET_HOST=http://gpu-node-01:8000 ./tests/locust/run_benchmark.sh
```

Скрипт последовательно прогоняет тест для concurrency = 1, 2, 4, 8, 16:

- длительность каждого прогона: **5 минут** (`RUN_DURATION=5m`);
- spawn-rate равен concurrency — все виртуальные пользователи стартуют
  одновременно, чтобы выйти на полку с первой секунды;
- между прогонами **пауза 30 секунд** (`COOLDOWN_SECONDS=30`), чтобы остатки
  KV-кеша и тепло GPU от предыдущего уровня не искажали следующий.

Перед началом скрипт обращается к `${TARGET_HOST}/health` и выходит с
ошибкой, если сервис недостижим, — это защищает от расхода 25 минут на
неправильно сконфигурированный таргет.

### Настраиваемые переменные

| Переменная           | По умолчанию              | Что делает                          |
|----------------------|---------------------------|-------------------------------------|
| `TARGET_HOST`        | `http://localhost:8000`   | Базовый URL сервиса                 |
| `RUN_DURATION`       | `5m`                      | Длительность одного прогона         |
| `COOLDOWN_SECONDS`   | `30`                      | Пауза между уровнями concurrency    |
| `CONCURRENCY_LEVELS` | `"1 2 4 8 16"`            | Список уровней (через пробел)       |

Например, быстрый sanity-прогон:

```bash
RUN_DURATION=1m CONCURRENCY_LEVELS="1 4" ./tests/locust/run_benchmark.sh
```

## Сводная таблица

После завершения всех прогонов соберите сводку:

```bash
python tests/locust/parse_results.py
```

Скрипт выведет Markdown-таблицу в stdout и параллельно сохранит её в
`results/locust/summary.md` и `results/locust/summary.csv`. Эти файлы
можно вставить прямо в текст ВКР.

## Состав нагрузки

`payloads.json` содержит 12 запросов с полями ровно в той форме, в которой
их принимает `app.api.schemas.GenerateRequest`:

- `user_id`, `message_content`, `grammar_error`, `explanation`,
  `llm_confidence`, `task_type`.
- Покрыты все четыре «контентных» типа задания: `grammar_choice`,
  `transformation`, `vocabulary_fill`, `matching` — по 3 запроса на тип
  (наименование `matching` соответствует поддерживаемому значению
  `TaskType`; в исходной формулировке задачи оно фигурировало как
  «functional_matching»).
- Категории ошибок разнообразны: Preposition, Tense, Article, Voice,
  Conditional, ReportedSpeech, WordChoice, Collocation.

Каждый виртуальный пользователь на каждой итерации выбирает payload
случайно — это сглаживает межзапросный разброс по сложности генерации.

## Что считается успехом

Только ответ, удовлетворяющий **обоим** условиям:

1. HTTP-код = 200.
2. В теле есть непустое поле `exercise`.

Ответ с `status == "fallback"` всё равно считается успешным (упражнение
выдано), но дополнительно учитывается в счётчике fallback. Любая
сетевая/таймаут/HTTP-5xx ошибка попадает в `failures` Locust и снижает
`success_rate`.

## Известные ограничения замера

1. **Сервис single-tenant.** README основного проекта (раздел 8) явно
   утверждает, что контейнер выполняет один `model.generate` за раз. На
   уровнях concurrency > 1 запросы в реальности встают в очередь на стороне
   uvicorn / FastAPI, а не выполняются параллельно на GPU. Поэтому ожидаемое
   поведение — RPS выходит на полку при concurrency, равной числу
   асинхронных воркеров uvicorn, а latency_p95 растёт почти линейно с
   concurrency. Это и есть тот эффект, который замер должен подтвердить.
2. **Таймаут 120 с.** Часть генераций (длинные `matching` с 8+ парами) может
   приближаться к таймауту — клиентский счётчик ошибок тогда покажет
   деградацию хвоста раньше, чем сервер сообщит 5xx.
3. **Колебания после холодного старта.** Первый запрос после `docker
   compose up` может прогревать кеш весов и токенайзер; запускайте бенчмарк
   только после того, как `/health` начнёт стабильно возвращать
   `loaded_models: ["qwen2.5-3b"]`.
4. **Сеть.** Клиент и сервер должны быть в одной L2-сети (или хотя бы в
   одном датацентре). RTT по интернету сместит p50 на десятки миллисекунд и
   замаскирует серверную составляющую.
5. **Шаблонный fallback дешёвый.** Высокий fallback_rate автоматически
   улучшит RPS и снизит latency, потому что fallback идёт мимо GPU. Всегда
   читайте RPS вместе с fallback_rate, а не по отдельности.
6. **Случайный выбор payload.** Распределение по `task_type` равномерное в
   пределе большого числа запросов, но на коротких прогонах (< 100 запросов
   на уровень) возможен перекос — лучше держать `RUN_DURATION` ≥ 5 минут.
7. **Никакой настройки tokenizers / dataloader.** Замер ничего не делает
   с серверной стороной, он только потребляет публичный API. Чтобы оценить
   тонкую настройку (batching, KV-cache reuse и т.п.), её нужно сначала
   реализовать в сервисе, а потом перезамерить.

## Полный прогон (готовая команда)

```bash
# 1. Установите зависимости (один раз)
pip install -r tests/locust/requirements.txt

# 2. Убедитесь, что сервис отвечает
curl -fsS http://localhost:8000/health

# 3. Прогоните бенчмарк (~28 минут: 5×5 мин + 4×30 с пауз)
./tests/locust/run_benchmark.sh

# 4. Соберите сводную таблицу
python tests/locust/parse_results.py
```

Готовая таблица — `results/locust/summary.md`.
